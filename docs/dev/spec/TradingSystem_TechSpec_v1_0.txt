SECTION 1 — SYSTEM OVERVIEW & ARCHITECTURE

◆ 1.1 System Purpose

This document is the complete technical specification for the AI-powered trading infrastructure. It defines every feature, component, data flow, integration, and behavioral rule for all three system modules. It serves as the authoritative reference for design, development, testing, and deployment.

The system is not a high-frequency trading bot or an indicator-based algorithm. It is an AI reasoning engine that thinks and analyzes markets the way an experienced discretionary trader does — combining macroeconomic intelligence, institutional flow data, and pure price action analysis under the SMC, Supply & Demand, and Wyckoff frameworks.

◆ 1.2 Design Principles

-   Reasoning over reaction: The AI reasons from structured knowledge, not pattern-matching indicators

-   Rules-first architecture: Every decision traces back to an explicit rule in the Master Rulebook

-   Pair-agnostic: The system applies identically to any instrument — no hardcoded pair logic

-   Style-configurable: Trading style (Scalping / Intraday / Swing / Positional) is a runtime setting

-   Zero hallucination design: AI retrieves rules from RAG — it does not speculate beyond what is documented

-   Full automation: Analysis → Execution → Management runs without manual intervention

-   Open source: Built on open-source components, publicly hostable, self-contained

◆ A.1 Overview

Module A is the brain of the system. It ingests real-time macroeconomic data and live chart data, processes both through a RAG-powered AI reasoning engine, and produces a structured, scored trade thesis for every active instrument on every analysis cycle. It does not place orders — it produces intelligence.

◆ A.2 Analysis Cycle

-   Module A runs on a configurable scheduled interval (default: every 4 hours, aligned with 4H candle close)

-   On each cycle: macro pipeline runs first → chart pipeline runs second → RAG engine synthesizes both → output is generated

-   If no setup is found: output is logged as NO SETUP and the system sleeps until the next cycle

-   If a setup is found: output is passed to Module B immediately

-   Multiple instruments can be analyzed per cycle — each produces an independent output

Macro Processing Rules

-   Currency macro bias is assessed independently for both currencies in every pair

-   DXY is always analyzed first on every cycle — its conclusion feeds directly into USD pair bias

-   COT data: Non-Commercial (speculative) positioning is the primary signal — large institutions and hedge funds

-   If COT shows extreme positioning (multi-year high/low) — flag as reversal risk, do not add in direction of extreme

-   High-impact news events within 30 minutes: system auto-flags, Module B will block execution during the window

-   Output: Macro Bias Score per currency (BULLISH / BEARISH / NEUTRAL) with supporting evidence chain

Hallucination Prevention Design

-   The AI is explicitly instructed in the system prompt that it may only reason from retrieved RAG documents and the real-time data provided

-   If a scenario is not covered by any retrieved document: output is NO SETUP — never a fabricated analysis

-   Every factor in the confluence score must be verifiably present in the live data — no assumed confluence

-   Each decision in the output cites the specific rule or document that supports it

-   Conflicting signals across timeframes are never resolved by assumption — they produce NO SETUP

-   The LLM temperature is set to 0 for deterministic, rule-consistent outputs

◆ B.1 Overview

Module B is triggered automatically when Module A produces a CONFIDENCE: HIGH output with a Grade A or A+ setup. It receives the structured output from Module A and executes the trade exactly as specified — applying the risk rules, calculating the precise lot size, and placing the order via the broker API. Module B does not re-analyze. It executes.

  -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  📌 Module B operates as a pure execution layer. It enforces the rules — it does not exercise judgment. If any pre-execution check fails, the trade is rejected and logged. No override is possible.

  -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

◆ B.2 Sub-Module B1 — Pre-Execution Validator

Before any order is placed, every condition in the validation checklist must pass. A single failure results in rejection. The validator runs sequentially — it stops and rejects on the first failure found.

  ---- ------------------------------- ------------------------------------------------------------------------- -----------
  #    Validation Check                Pass Condition                                                            On Fail

  1    Confluence Score                Must meet minimum threshold for active trading style                      REJECT

  2    Setup Grade                     A+ or A grade (B grade configurable via dashboard)                        REJECT

  3    Higher Timeframe Alignment      1W and 1D must both agree with trade direction                            REJECT

  4    News Event Lockout              No high-impact news within 30 minutes (45 min in Scalping mode)           REJECT

  5    Session Filter                  Current session must be enabled in dashboard settings                     REJECT

  6    Existing Position — Same Pair   No open trade or pending order on the same instrument                     REJECT

  7    Correlated Pair Exposure        Max 1 trade per correlated group — take the strongest setup only          REJECT

  8    Maximum Concurrent Trades       Open trade count must be below configured maximum (default: 3)            QUEUE

  9    Daily Loss Limit                Daily drawdown must be below 3% limit                                     LOCK

  10   Weekly Drawdown Limit           Weekly drawdown must be below 5% limit                                    PAUSE

  11   Spread Check                    Current spread must not exceed 2x normal average (1.5x in Scalping)       REJECT

  12   Minimum R:R Confirmation        R:R must meet minimum for active trading style (1:2 scalp, 1:3 others)    REJECT

  13   Weekend/Day Filter              No entries after Friday 12:00 UTC (intraday/scalp) or 14:00 UTC (swing)   REJECT

◆ B.3 Sub-Module B2 — Position Sizing Engine

Lot Size Calculation Formula

+---------------------------------------------------------------------------+
| Lot Size = (Account Balance × Risk %) ÷ (SL Distance in pips × Pip Value) |
|                                                                           |
| Where:                                                                    |
|                                                                           |
| Account Balance = Current live balance (fetched from broker API)          |
|                                                                           |
| Risk % = Style/grade-based risk (A+/A = 1%, B = 0.5%)                     |
|                                                                           |
| SL Distance = |Entry Price - Stop Loss Price| converted to pips           |
|                                                                           |
| Pip Value = Per-pip value for the instrument (fetched from broker)        |
|                                                                           |
| Example:                                                                  |
|                                                                           |
| Balance: $10,000 | Risk: 1% ($100) | SL: 25 pips | Pip Value: $10/lot     |
|                                                                           |
| Lot Size = $100 ÷ (25 × $10) = $100 ÷ $250 = 0.40 lots                    |
+---------------------------------------------------------------------------+

-   Lot size is calculated fresh on every trade using current live account balance — never static

-   Fractional lots are used to achieve precise risk amounts (e.g. 0.43 lots)

-   Minimum lot size floor: 0.01 lots — if calculated size is below floor, trade is rejected

-   Maximum lot size ceiling: configurable per instrument (prevents outsized exposure on exotic pairs)

◆ B.4 Sub-Module B3 — Hybrid Execution Mode

The system supports two execution modes selectable from the dashboard. Both modes run the identical full analysis pipeline — same Module A cycle, same confirmation rules, same confluence requirements, same RAG validation. The ONLY difference is the execution mechanism after the setup is validated and the entry zone is confirmed.

  -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  📌 Same intelligence. Same rules. Same confirmations. Different trigger mechanism. The mode controls HOW the order is placed — not WHAT is analyzed or validated before it.

  -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------

  ---------------------- -------------------------------------------------------------------------------------------------- -------------------------------------------------------------------------------------------
                         LIMIT MODE                                                                                         INSTANT MODE

  Analysis Pipeline      Full Module A cycle — macro, chart, RAG, confirmations, confluence                                 Full Module A cycle — macro, chart, RAG, confirmations, confluence

  Confirmation Rules     Applied in full — retrieved from RAG documents                                                     Applied in full — retrieved from RAG documents

  Order Type             Limit Order — placed at broker at the confirmed entry price immediately after analysis completes   Market Order — fired the moment price touches the confirmed entry level in real-time

  Who Watches Price      Broker — holds the limit order and fills automatically when price arrives                          Module C Go goroutine — monitors every tick and fires on touch

  Fill Price             Exact entry price guaranteed — zero slippage by design                                             At-touch market price — minimal slippage, fired instantly

  Speed Requirement      Low — order placed in advance, no real-time monitoring needed                                      High — Go goroutine tick monitoring, sub-millisecond trigger

  Best For               Clean zone entries where price is expected to retrace calmly into the zone                         Sweep entries, fast-moving price, entries where pre-placing risks a bad fill on overshoot

  Pending Order Expiry   Configurable TTL (time-to-live) — limit cancelled if not filled within X candles                   No pending order — goroutine watches until entry fires or setup invalidates
  ---------------------- -------------------------------------------------------------------------------------------------- -------------------------------------------------------------------------------------------

Dashboard Execution Mode Controls

-   Global default: trader sets preferred mode once — applies to all pairs and setups

-   Per-pair override: EURUSD can run Limit while XAUUSD runs Instant simultaneously

-   Module A recommendation: the analysis output includes a recommended execution mode based on setup type — trader can accept or override from dashboard before next cycle

-   Mode changes take effect on the next analysis cycle — no mid-setup switching

Limit Order — Pending Management Rules

-   Limit order is placed immediately when Module B completes execution validation

-   Entry price: OTE zone (62–79% of OB) or top/bottom of confirmed SnD zone

-   Stop loss and all three TP levels are attached to the order at placement time

-   TTL (Time-To-Live): configurable per trading style — e.g. Intraday: cancel if unfilled after 1 session, Swing: cancel after 3 days

-   If market structure breaks before fill (1D BOS against trade direction) — limit order cancelled automatically

-   If a new Module A cycle runs and the thesis changes — pending limit is cancelled and replaced or removed

Instant Mode — Go Goroutine Trigger Rules

-   After Module B validates the setup, it arms Module C with: entry zone, SL, TP levels, lot size, direction, and invalidation conditions

-   Module C Go goroutine begins tick monitoring immediately on the confirmed instrument

-   Trigger condition: ask price (for longs) or bid price (for shorts) touches the confirmed entry level

-   On trigger: market order fires immediately via broker API — no delay, no confirmation step

-   If price overshoots entry zone without reversal signal — goroutine applies overshoot tolerance rule (configurable, default: entry zone width × 1.5)

-   If setup invalidation conditions are met before trigger — goroutine disarms, no order placed, logged as EXPIRED

◆ C.1 Overview

Module C activates the moment Module B confirms an order is FILLED. From that point, Module C owns the trade entirely — from entry to final exit. It monitors price in real time, manages stop loss adjustments, executes partial closes at take profit levels, handles end-of-period protocols based on the active trading style, invalidates trades when the thesis breaks, and automatically journals every closed trade with full detail.

  --------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  ✅ Module B hands the trade to Module C. Module C never hands it back. The trade lifecycle ends when Module C logs the final close. No manual intervention at any point.

  --------------------------------------------------------------------------------------------------------------------------------------------------------------------------

◆ C.2 Sub-Module C1 — Trade Monitoring Engine

Module C runs two parallel monitoring tracks for every active instrument: the pre-fill track (Instant Mode only) and the post-fill management track (both modes). Both are implemented as Go goroutines running concurrently.

Pre-Fill Track — Instant Mode Goroutine

-   Activated when Module B arms Module C after a validated Instant Mode setup

-   Go goroutine monitors every incoming tick on the instrument via MT5 price feed

-   Trigger condition: ask price (longs) or bid price (shorts) touches confirmed entry level

-   On trigger: market order fired immediately via broker API — sub-millisecond from touch to order

-   Overshoot tolerance: if price pierces entry zone by more than configured tolerance without reversal — goroutine disarms, logs EXPIRED, notifies trader

-   Invalidation watch: if structure breaks before trigger (1D BOS against thesis) — goroutine disarms automatically

-   On fill confirmed: pre-fill goroutine terminates, post-fill management goroutine activates

Post-Fill Track — Management Monitoring

  ----------------------------- ----------------------------- -----------------------------------------------------
  Monitor                       Frequency                     Action Triggers

  Price vs SL / TP levels       Every tick / every 1 minute   SL hit → close. TP hit → partial close + SL adjust.

  Structure change on 4H + 1D   Every candle close            BOS in opposite direction → tighten SL

  New macro / news events       Every 15 minutes              High-impact event vs position direction → evaluate

  COT weekly update             Friday release                COT flip against position → tighten SL, flag trade

  Correlated pair behavior      Every 4H candle close         Correlated pair confirms opposite → tighten SL

  Session open / close          Time-based triggers           EOD / end-of-session protocols activate

  Break-even trigger            Every tick                    TP1 level reached → SL moves to entry + buffer
  ----------------------------- ----------------------------- -----------------------------------------------------

◆ C.3 Sub-Module C2 — Stop Loss Management

Break-Even Rules

-   Trigger: Price reaches TP1 level → SL immediately moves to entry price + spread buffer (2–3 pips)

-   Scalping exception: BE triggers when price moves 60% toward TP1 on 15M chart

-   Intraday time-based exception: If TP1 not reached within 3 hours → tighten SL to 50% of original risk

-   Break-even is strictly one-directional — SL only moves in profit direction, never pulled back

-   If price returns to BE level after trigger — trade closes at break-even. Logged as BREAK-EVEN, not a loss.

Trailing Stop — Style-Adaptive

  ----------------- --------------------------------------------------------------------- -------------------------
  Trading Style     Trail Mechanism                                                       Recalculation Frequency

  Scalping          Trail to most recent 15M swing high/low                               Every 15M candle close

  Intraday          Trail to 1H swing high/low. Switch to 4H during London/NY overlap.    Every 1H candle close

  Swing             Trail to 4H swing high/low after TP1. Switch to 1D swing after TP2.   Every 4H candle close

  Positional        Trail to 1D swing high/low after TP1. Switch to 1W swing after TP2.   Every daily close
  ----------------- --------------------------------------------------------------------- -------------------------

◆ C.4 Sub-Module C3 — Take Profit Execution

  ------------ ------------------ ------------------------------------------------------------ -------------------------------
  Target       Position Closed    Level Basis                                                  Post-TP SL Action

  TP1          40% (Scalp: 60%)   Nearest liquidity pool / minor resistance / previous swing   Move SL to break-even

  TP2          30% (Scalp: 40%)   Major S&D zone / HTF key level / 1D structure target         Trail SL to TP1 level

  TP3 Runner   30% (Scalp: N/A)   1W key level / final macro target / trailed until hit        Trail on style-appropriate TF
  ------------ ------------------ ------------------------------------------------------------ -------------------------------

-   Each partial close is executed as a separate broker order — remaining position stays open with updated SL

-   After each partial close: lot size updates, SL adjusts per rules, notification sent, journal entry updated

-   Scalping mode: No TP3/runner — full position closed by TP2 at the latest or session end

◆ C.5 Sub-Module C4 — End-of-Period Protocols

Scalping — End of Session

-   All scalp trades forced-closed before session end regardless of status

-   Maximum trade duration: 2 hours from entry — hard time-based close

-   No carry to next session under any circumstances

Intraday — 16:30 UTC Daily Protocol

  ---------------------------------------------------------------------- -------------------------------------------------
  Trade Status at 16:30 UTC                                              Automated Action

  In profit — TP1 hit — SL at break-even or better                       Close remaining position at market. Log WIN.

  In profit — TP1 not yet reached                                        Close at market. Log partial WIN.

  At break-even                                                          Close at market. Log BREAK-EVEN.

  In loss — SL not yet hit                                               Close at market. Log LOSS. No overnight hold.

  Overnight carry exception (toggle ON + SL at BE + no overnight news)   Hold position. Re-evaluate at next London open.
  ---------------------------------------------------------------------- -------------------------------------------------

Swing — Friday 16:00 UTC Protocol

-   SL at break-even or better AND overnight carry toggle ON → may hold over weekend

-   SL still in original loss territory → close at market regardless of unrealised P&L

-   Re-evaluate all swing trades at Monday 07:00 UTC before London open

Positional — Weekly Review (Friday 15:00 UTC)

-   No forced closure — positional trades run on their own structural logic

-   System assesses weekly candle close — if 1W close invalidates the thesis → close Monday open

-   Weekend hold is standard and expected — SL placement accounts for weekend gap risk

◆ C.6 Sub-Module C5 — Trade Invalidation Engine

  ------------------------------------------------------------------------------------------------------------
  ⚠ If the reason the trade was entered no longer exists — the trade no longer exists. The system closes it.

  4H BOS confirmed in opposite direction while in trade       Tighten SL to most recent 4H swing. Prepare for exit.

  High-impact news surprise against position direction        Hold unless SL triggers. If 4H structure breaks → close.

  COT positioning flips against open trade on weekly update   Flag trade. Tighten SL. No further position additions.
