# Execution & Gateway Codebase Audit

## 1. Execution Service (Module B) Current State
**Current Implementation:**
After auditing [src/execution/internal/executor/executor.go](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/execution/internal/executor/executor.go) and [grpc_server.go](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/gateway/internal/server/grpc_server.go), the system currently processes orders via [ExecuteTrade](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/execution/internal/server/grpc_server.go#194-388). 
If the execution mode is Limit, it places the Limit Order natively at the MT5 broker. 
If the execution mode is Instant, it simply generates a `WatcherID` and records an intent to watch the setup, explicitly commenting: *"Module C is responsible for watching ticks and firing the market order... The arming handoff to Module C happens via gRPC... Module B's job ends at recording the intent."*
It completely lacks background tracking logic for Instant orders.

**Missing Capabilities for 100% Native Execution:**
1. **Live Price Feed Subscription:** [src/execution/internal/broker/mt5/bridge.go](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/execution/internal/broker/mt5/bridge.go) lacks a `StreamTicks` or `GetLivePrice` method to actively monitor the `entry_price` bounds.
2. **Background Goroutine Spawning:** [executor.go](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/execution/internal/executor/executor.go) needs a true [armInstant](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/execution/internal/executor/executor.go#107-139) function that spins up `go e.watchPrice(ctx, order)` instead of instantly returning successful.
3. **Gateway Call-Back Trigger:** Once the tick reaches the entry zone, Execution needs a gRPC client to call back to the Gateway (`ConfirmSetup`) and, upon True, fire the market order natively.

## 2. Gateway Service Current State
**Current Implementation:**
After auditing [src/gateway/internal/pipeline/orchestrator.go](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/gateway/internal/pipeline/orchestrator.go) and [routing/router.go](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/gateway/internal/routing/router.go), the Gateway orchestrates the entire intelligence loop seamlessly via [RunCycle()](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/gateway/internal/server/grpc_server.go#110-145). It routes successfully processor-approved trades to Execution using `ExecutionGRPCAdapter.Execute()`.

**Missing Capabilities for Orchestration Loop:**
1. **Targeted TA Confirmation Pulse:** The Gateway only has a monolithic [RunCycle()](file://wsl.localhost/Ubuntu-24.04/home/softverse/eTradie/src/gateway/internal/server/grpc_server.go#110-145) that takes minutes due to Macro/RAG/Processor. It needs a fast, dedicated endpoint (e.g., `ConfirmInstantSetup`) exposed via gRPC/HTTP that Execution can call.
2. **Bypass RAG/Macro:** This new endpoint must ONLY call the TA Engine, verify the specific `AnalysisID` setup, and check if `ltf_confirmation == True` (Session Timing + RTO touch).
3. **Signal Execution:** Gateway returns a boolean. If True, Execution completes the market order execution and updates the DB.

## 3. Next Implementation Steps
1. **Broker Bridge Updates:** Update Python MT5 Engine and Go Broker Bridge to support querying live tick prices.
2. **Gateway Pulse Endpoint:** Add a `ConfirmSetup` RPC to the Gateway service.
3. **Execution Native Watcher:** Rewrite `executor.armInstant` to actually spawn a monitoring goroutine, poll the live price, and execute the full call-back loop.

## 4. Setup Type Tracing Audit (Resolved)
**Issue Identified:** The `setup_type` field generated by the TA Engine was successfully retrieved by the Gateway `Processor` but was being dropped during the handoff to the Execution engine and subsequently the Management engine.
**Resolution:** 
1. **Protobufs:** Updated `ExecuteTradeRequest` (Execution) and `NotifyExecutionCompletedRequest` (Gateway) to natively include `setup_type`.
2. **Execution Models:** Added `SetupType` to `TradeRequest` and `Order` structs.
3. **Gateway & Execution Mappers:** Threaded the `SetupType` from `ProcessorOutput` -> `ExecuteTradeRequest` -> `Order` -> `NotifyExecutionCompletedRequest` -> `RegisterFilledTradeRequest`.
**Status:** The data pipeline is now 100% loss-less end-to-end.
